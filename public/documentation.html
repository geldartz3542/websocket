Digisaka WebSocket Server

Production-ready, Pusher-compatible WebSocket server for real-time communication between the Laravel API, Vue admin panel, and Flutter mobile app.

ℹ How it works

Clients (Vue, Flutter, browser) connect via WebSocket and subscribe to channels.

Laravel pushes events through the REST API (POST /api/trigger).

The server broadcasts events to all subscribers of the targeted channel(s).

Protocol Reference

The server follows a Pusher-compatible event protocol. All messages are JSON.

Connection Flow

Client Server │ │ │──── WS Connect ───────────────────►│ │ │ │◄── connection_established ────────│ (returns socket_id) │ │ │──── identify (user_id) ──────────►│ │◄── identified ────────────────────│ │ │ │──── subscribe (channel) ─────────►│ │◄── subscription_succeeded ────────│ │ │ │◄── events from server ───────────│ (via REST API trigger) │ │ │──── unsubscribe (channel) ───────►│ │──── close ───────────────────────►│
1
Connect

Client opens a WebSocket connection. Server replies with connection_established containing a unique socket_id.

2
Identify (optional)

Client sends identify with a user_id to associate the socket with an app user for direct messaging.

3
Subscribe

Client sends subscribe with a channel name. Private/presence channels require an auth token.

4
Receive events

Events triggered by the server (via REST API) or other clients are pushed to the subscriber.

Client → Server Events

WS identify

Associate a user ID with this connection.

json
{ "event": "identify", "user_id": "42" }

Response:

json
{ "event": "identified", "data": "{\"user_id\":\"42\",\"socket_id\":\"123456.7890123\"}" }

WS subscribe

Join a channel. For private/presence channels, include an auth token.

Public
Private
Presence
json
{ "event": "subscribe", "channel": "alerts" }
json
{
  "event": "subscribe",
  "channel": "private-user.5",
  "auth": "digisaka-ws-key-2026:a1b2c3d4e5f6..."
}
json
{
  "event": "subscribe",
  "channel": "presence-chat.1",
  "auth": "digisaka-ws-key-2026:a1b2c3d4e5f6...",
  "channel_data": { "user_id": "5", "user_info": { "name": "Juan" } }

WS unsubscribe

json
{ "event": "unsubscribe", "channel": "alerts" }

WS message (to channel)

Broadcast a message to all other subscribers of a channel.

json
{ "event": "message", "channel": "alerts", "message": "Hello everyone" }

WS message (to users)

Send a private message to specific users.

json
{ "event": "message", "target_users": ["5", "12"], "message": { "text": "Hi" } }

WS client-* events

Events prefixed with client- are broadcast to the channel (excluding sender). Follows Pusher convention.

json
{ "event": "client-typing", "channel": "presence-chat.1", "data": { "user": "John" } }

WS pusher:ping

Application-level heartbeat. Server replies with pusher:pong.

json
{ "event": "pusher:ping" }

EventDescription

subscription_succeededConfirms channel join. Presence channels include member list.

(custom)Any event triggered via the REST API (e.g. new-alert, task-updated).

Channel Types

public Public Channels

Any name without a private- or presence- prefix. No authentication required.

examples
alerts
news
demo-trials
farm-updates

Any client can subscribe and receive events.

private- Private Channels

Prefixed with private-. Requires a valid auth token to subscribe.

examples
private-user.42
private-admin.notifications
private-farm.15

Auth token calculation:

text
signature = HMAC-SHA256(APP_SECRET, "socketId:channelName")
token     = "APP_KEY:signature"

presence- Presence Channels

Prefixed with presence-. Requires auth + member info (channel_data).

examples
presence-chat.room.1
presence-editors.doc.5
  • Member tracking — server maintains who is in the channel
  • member_added event when someone joins
  • member_removed event when someone leaves
  • subscription_succeeded includes the current member list

REST API Reference

All authenticated endpoints require these headers:

headers
X-App-Key: <your APP_KEY>
X-App-Signature: <your APP_SECRET>
ℹ Auth modes

Simple match — X-App-Signature equals the raw secret (convenient for trusted server-to-server calls).

HMAC-SHA256 — X-App-Signature is HMAC-SHA256(secret, requestBody) for stricter security.

POST /api/trigger Push events to channel(s) ▶

The primary endpoint for server-side event broadcasting.

Request body

FieldTypeRequiredDescription
channelstring*Single channel name
channelsstring[]*Multiple channel names
eventstringyesEvent name
dataanynoEvent payload
socket_idstringnoExclude this socket (prevent echo)

* Either channel or channels is required.

Example

bash
curl -X POST http://localhost:6001/api/trigger \
  -H "Content-Type: application/json" \
  -H "X-App-Key: digisaka-ws-key-2026" \
  -H "X-App-Signature: digisaka-ws-secret-change-me-in-production" \
  -d '{
    "channels": ["alerts", "private-admin"],
    "event": "new-alert",
    "data": { "farm_id": 5, "type": "drought", "severity": "high" }
  }'

Response

json
{ "success": true, "channels": ["alerts", "private-admin"], "recipients": 8 }

Usage — WebSocketService

php
use App\Services\WebSocketService;

Trigger events on channels

php
// Single channel
WebSocketService::trigger('alerts', 'new-alert', [
    'farm_id' => 5,
    'type' => 'drought',
    'message' => 'Low moisture detected',
]);

// Multiple channels
WebSocketService::trigger(
    ['alerts', 'private-user.12', 'private-user.34'],
    'task-assigned',
    ['task_id' => 99, 'title' => 'Inspect Farm 7']
);

// Exclude sender (prevent echo)
WebSocketService::trigger('chat.room.1', 'new-message', $data, $request->socket_id);

Send directly to users

php
WebSocketService::toUsers([5, 12], 'notification', [
    'title' => 'Approval needed',
    'body' => 'Demo trial #42 is waiting for your approval.',
    'url' => '/approval/42',
]);

Authorize channels

php
$auth = WebSocketService::authorizeChannel(
    $request->socket_id,
    $request->channel_name,
    $channelData // for presence channels
);
return response()->json($auth);

Get stats

php
$stats = WebSocketService::stats();
$channels = WebSocketService::channels();

Real-World Examples

In a Controller

php
public function storeAlert(Request $request)
{
    $alert = Alert::create($request->validated());

    WebSocketService::trigger('alerts', 'new-alert', [
        'id'       => $alert->id,
        'type'     => $alert->alert_type,
        'message'  => $alert->alert_message,
        'severity' => $alert->severity,
    ]);

    return response()->json(['success' => true, 'data' => $alert]);
}

In a Queued Job

php
class NotifyApprovers implements ShouldQueue
{
    public function handle()
    {
        $approverIds = User::role('approver')->pluck('id')->toArray();

        WebSocketService::toUsers($approverIds, 'pending-approval', [
            'count' => DemoTrial::where('status', 'pending')->count(),
        ]);
    }
}

In an Observer

php
class DemoTrialObserver
{
    public function updated(DemoTrial $trial)
    {
        if ($trial->isDirty('status')) {
            WebSocketService::trigger(
                "private-user.{$trial->user_id}",
                'trial-status-changed',
                [
                    'trial_id'   => $trial->id,
                    'old_status' => $trial->getOriginal('status'),
                    'new_status' => $trial->status,
                ]
            );
        }
    }
}

Lifecycle

ℹ Automatic lifecycle

Disconnects when app goes to background. Reconnects on resume.

Resubscribes to all channels after reconnecting.

Uses exponential backoff: 1s → 2s → 4s → ... → 30s max.

Cleanup

dart
ws.disconnect(); // stops auto-reconnect
ws.dispose();    // full cleanup (on logout)

const ws = new WebSocket('wss://websocket.digisaka.app');

ws.onmessage = (event) => {
  const msg = JSON.parse(event.data);

  }
};

async def main():

Security

Best Practices

  • Change APP_SECRET from the default before deploying (use 64+ random chars)
  • Set ALLOWED_ORIGINS to actual domains, not *
  • Use TLS — terminate at nginx/Cloudflare or set SSL_CERT_PATH
  • Rate limiting is enforced per-connection (default: 60 msg/min)
  • IP limits prevent single-IP abuse (default: 50 connections/IP)
  • Never expose APP_SECRET in client-side code
  • Channel auth should always go through your Laravel backend
⚠ Important

Private and presence channel auth tokens must be generated server-side. The Flutter/Vue client should request auth from your Laravel API, which calls WebSocketService::authorizeChannel().
